Modul 4 — Arhitectură Hardware

Lecția 4.2 — GPU: Geometrie, Shadere și Pixeli

Render-gu imaginu: De la vârf virtual la pixel pe ecran. Miliarde operații paralel.

Începe Lecția

Introducere

GPU (Graphics Processing Unit): Specialista paralelismului. Mii de thread-uri mici executa simultan pixeli diferite.

Diferență CPU vs GPU: CPU = putere de calcul serială (IPC mare per core). GPU = bandwidth masiv (mii calcule paralel pe pixeli).

PS5 GPU: 36 compute units, 2304 stream processors. 10.28 TFLOPS (tera-floating-point-operații/sec). AMD RDNA arch.

Teorie Structurată

4.1 — Rendering Pipeline

Input: Vertex buffer (liste de (x,y,z,w,u,v) coordonate). Triangle definit de 3 vertice.

Vertex shader: GPU lanseaza 1 shader per vertex (paralel). Transforma vertex coords din model space la screen space. Mii de vertice simultane.

Rasterization: Determina care pixeli curg în interioru triunghi. Hardware scanline.

Fragment shader: GPU lanseaza 1 shader per pixel in triung. Calculeaza culoare (texture lookup, lighting, etc). PARALEL pe toate pixeli.

Output: Framebuffer. Pixel culori => VRAM => monitor 60 Hz.

4.2 — Warp & Thread Organization

Warp: Grup 32 thread-uri (AMD) executa aceeași instrucție pe date diferite (SIMD).

Wave (AMD): 64 thread-uri per compute unit. Lanseaza mii simultan pe 36 CU = 36×64 = 2304 paralel.

Execution model: Thread-uri gândesc local (care pixel? care vertex?). Hardware synchronizeaza operații globale (atomic, barrier).

4.3 — Shaders și Instruction Latency

Vertex shader: ~10-50 instrucțiuni per vertex. Matrici transform (4×4), projections.

Fragment shader: ~50-500 instrucțiuni per pixel. Textură lookups (~4 ciclu latență), lighting math, depth test.

Latency hiding: Compute unit are 1000+ thread-uri în flight. Cand 1 warp stall (memory latency), executor switchează la alt warp. Zero stall daca suficiente waveleți!

4.4 — Texture Cache & Bandwidth

Texture unit: Specialista cache pentru imagini. 64K L1-texture cache per CU. Rapid texture lookups (ex. Samplu normal map pe 2304 pixeli simultane).

Bandwidth: GPU → VRAM = 448 GB/sec (PS5 GDDR6). CPU → VRAM = 448 GB/sec shared. Throughput = fiecare shader lanse output + fetch input rapid.

4.5 — GPU Limitations și Occupancy

Occupancy: % CU compute units efectiv cu waveleți. Ocupanță bună 70-90%. Slabă <25% = stalls.

Bottleneck: ALU-bound shaders (calcul intens) vs memory-bound (texture lookups, G-buffer access).

PS5 limite: 10.28 TFLOPS. Dar dacă memory-bound, throughput AI (arithmetic intensity, operații per byte fetched) control real perf.

Legătura Fizică – Informatică

Fizică: Mii de tranzistori comutează în paralel pe pixeli diferite. GDDR6 memorie transferu 14 GBps bandwidth.

Informatică: Shader code transformat în maschinsprog. Fiecare pixel = independent task. Procesare masiv paralelă.

Aplicare Directă

Exemplu: Render Frame 1080p 60 FPS

Framebuffer: 1920 × 1080 = 2,073,600 pixeli. 4 bytes RGBA = 8.3 MB per frame.

60 FPS: 60 × 60 FPS = 60 frames/sec. Throughput pixel: 2 × 60/sec × 60 = 124.4 megapixel/sec.

GPU task: Lanseaza 2M fragment shader threads simultane (paralle pe waveleți). Fiecare shader: texture fetch + lighting = ~200 instrucțiuni.

Bandwidth: 2M pixeli × 8 bytes output + textură lookups ~1KB per pixel = 2 GB/sec effective utilization din 448 GB/sec max (0.4% utilizare!).

→ GPU super ocup. Vertex shader ~ 1M thread (mii triungi, 3M vertex total). parallelism masiv.

Exemplu Real: Unreal Engine 5 Exemplo PS5

Scenario: Rendering 4K resolución (3840×2160) cu 60 FPS, Ray-traced shadows.

Challenge: 8.3M pixeli × 60 = 498M pixeli/sec. Cu ray-trace, ~100 instruc/pixel = 49 miliarde instruc/sec = 4.7 TFLOPS.

PS5 utilization: 10.28 TFLOPS / 4.7 = 45% occupanță. Plenty of headroom. Deci putem adaugă mai mulți pixeli (post-effects) sau mai complex shader.

Probleme Frecvente

⚠️ De ce 4K pe PS5 = problema?

4K = 4× pixeli decât 1080p. Framebuffer 3× mai mare. Bandwidth 3× plus. Shaders ~3× mai lungă. Score IPC cade dacă overhead mari. 30 FPS normal, 60 FPS rare.

⚠️ De ce ray-tracing scump per pixel?

Ray-trace = 5-10 instruc per pixel vs 100 per standard shader. GPU-ul trebuie să facă milioane calcule simultaneas. Thermal power limitează.

Recapitulare

  • GPU: Specialista paralelismului. Mii thread-uri pe pixeli simultan.
  • Pipeline: Vertex → Rasterize → Fragment shader → Framebuffer.
  • Compute unit: 36 CU × 64 thread-uri = 2304 paralel.
  • Latency hiding: Wavelet-uri switch cand memory stall.
  • PS5: 10.28 TFLOPS, 448 GB/sec GDDR6, 10.7 GB texture cache.
  • Shader optimization = key pentru 60 FPS + 4K (dificil pe PS5).

Exercițiu Aplicat

🧠 Exercițiu: Shader Throughput Analiza

Cerință: Fragment shader: 300 instrucțiuni (float, ALU, texture lookups). Fiecare lookup latență 4 ciclu. Clock 1 GHz (GPU = slower ALU, latenci masqué).

  1. Instrucțiuni critical path (latență dominator)?
  2. Waveleți necesare pentru hide latență texture (4 ciclu)?
  3. Throughput pixel/sec dacă 64 waveleți per CU, 36 CU?
Arată rezolvarea

1. Critical path: Texture lookup = 4 ciclu dominator. ALU e mai rapid (1-2 ciclu). Optimal: launch next wavelet cand current waitem.

2. Waveleți hide latență: 4 ciclu latență / (1 ciclu per ALU) = need 4 waveleți în flight (simplice). Real cu dependencies: 8-16 waveleți.

3. Throughput: 64 waveleți per CU × 36 CU = 2304 thread-uri simultane. Daca fiecare shader = 10 ciclu latență (texture 4 + ALU 6), throughput = 2304 × 100 MHz = 230 mpixel/sec.

Comparație: 1080p 60 FPS = 124 mpixel/sec. Shader can handle 1.8× load. Good ocupanță!
