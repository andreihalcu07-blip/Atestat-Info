Modul 3 — Logică Digitală

Lecția 3.5 — Construcția Logică a unui Procesor

De la biți și porți logice la ALU care calculează. Cum se pune totul laolaltă.

Începe Lecția

Introducere

CPU: Assemblare de logică digitală. Porți logice → ALU (Arithmetic Logic Unit) → Control logic → Registre → Memorie interface.

Scopul CPU: Ia instrucție din memorie, decode-o, execute operație (ADD, LOAD, STORE, JUMP), ține rezultat în registre.

PS5 CPU = 8 ARM cores, fiecare cu propriul decode logic, ALU, cache. Miliarde porți logice coordinate pe clock.

Teorie Structurată

3.1 — ALU (Arithmetic Logic Unit)

Meniu operații: ADD, SUB, AND, OR, XOR, SHIFT, CMP.

Implementare: Cascadă adder-itori cu propagare carry. 64 biți ALU = 64 1-bit adder-uri în serie (carry ripple) sau paralel (carry lookahead = mai rapid).

Exemplu ADD: 32-bit adder: XOR porți (biți), carry propagare la fiecare bit, output = sumă 32-bit + carry out (overflow flag).

Selector: MUX selecteaza operație (ADD vs OR vs etc.). OpCode (3-5 biți) selecteaza ce ALU operare.

3.2 — Registre și Register File

Register file: 32× 64-bit flip-flop registry. Fiecare registru (RAX, RBX, ..., R31) ține 64 biți.

Multiplexing: Instrucție specifica "read RAX, write a RBX". MUX selecteaza care bit din register file să scoată.

Timing: Read latență ~1 ciclu. Write latență 1 ciclu cu clock edge. Registre sunt SRAM = ultra-rapid.

3.3 — Control Logic și Decoder

Instruction decoder: Ia 32-bit OpCode. Decode-aza în semnale control: ALU opcode, register selectori, memory read/write, branch condition.

Example: OpCode = 0x8B45 (MOV RBX, [RAP+offset]). Decoder vede: memory load, RAP + offset calcul în ALU, rezultat → RBX.

Microcode ROM: Tabel de lookup. OpCode → microinstrucții. Modern: direct logic combinațional (decode hardware).

3.4 — Data Path: Fetch-Decode-Execute

Fetch: PC (program counter) → adresa memorie. Citesc instrucție 32-bit din cache/RAM.

Decode: Instrucție în control semnale. Selecteaza registre sursa, ALU opcode, destinație.

Execute: Citesc din registru file (1 ciclu), ALU calculează (1 ciclu), rezultat în registru destinație (1 ciclu).

Latență típic: 3-5 ciclu per instrucție (legacy). Pipeline modern reduce la 1 ciclu/instrucție (throughput).

3.5 — Pipelining și Parallélism

Pipeline: Fetch, Decode, Execute, Memory, Writeback runeaza simultan pe instrucții diferite.

Exemplu: Ciclu 1: Instr1 fetch. Ciclu 2: Instr1 decode, Instr2 fetch. Ciclu 3: Instr1 execute, Instr2 decode, Instr3 fetch.

Throughput: 1 instrucție/ciclu (daca fără pipeline stalls). 3.5 GHz × 1 instrucție/ciclu = 3.5 miliarde instrucții/sec.

Hazard-uri: Data dependency (instr2 citire rezultat instr1 care nu-i gata). Branch misprediction. Cache miss.

Legătura Fizică – Informatică

Fizică: Porți logice comutează în nanosecunde. Registre memorează în nanoSec. Clock coordonează timing.

Informatică: Ciclu de fetch-decode-execute se repetă miliarde ori/sec, dând iluzia "program care rulează".

Aplicare Directă

Exemplu: Execuție MOV RBX, [RAX]

Instrucție: Mut conținut memorie de la adresa RAX în registrul RBX.

Ciclu 1 (Fetch): PC = instrucție address. Citesc 4 bytes = MOV opcode.

Ciclu 2 (Decode): Decoder vede MOV. Register selector = RAX. Operație ALU = ADD (RAX+0 offset).

Ciclu 3 (Execute): ALU calculează: RAX valoare (ex. 0x1000). Trimite adresa memorie.

Ciclu 4 (Memory): L1 cache primeste 0x1000. Citire 8 bytes de data.

Ciclu 5 (Writeback): Data → RBX registru via register file MUX selectare.

Total latență: 5 ciclu. Dar pipeline: Instr2 în ciclu 2, Instr3 în ciclu 3, etc. Output: 1 instrucție/ciclu throughput.

Exemplu Real: ARM64 Instrucție cu Branch

Instrucție MOV: mov x1, #0x1000 (move constant 0x1000 în registru x1).

Branch: cmp x1, x0 (compare x1 cu x0). bne .skip_func (branch if not equal).

Problem: CPU nu stie dacă sare până cmp se termina. Decode trebuie specesa că bne e conditional.

Branch prediction: CPU ghiceste dacă sare. Fetch instr de la skip_func address. Daca ghicire greș, pipeline flush = 10+ ciclu penalty!

Penalty impact: 3.5 GHz. Misprediction = 10 ciclu penalty. 1 misprediction la 20 instrucții = 0.5 ciclu latență medie. Performance = 70% din teoretic.

Probleme Frecvente

⚠️ De ce pipeline hazard-urile problematice?

Data dependency: Instrucție n+1 trebuie data din n care nu-i gata. Trebuie stall (trage) ciclu. Performance cade.

⚠️ De ce branch misprediction e costisitor?

Pipeline flush = 10-15 ciclu wasted. Per instr rata 20%, misprediction rata 10% → 1 ciclu penalty medie. Performance -30%+.

Recapitulare

  • ALU: Cascadă de porți logice. ADD, SUB, AND, OR, XOR, SHIFT.
  • Register file: 32× 64-bit flip-flop registru. Citire/scriere SRAM-speed.
  • Control decoder: OpCode → microinstrucții → semnale control ALU, registre, memorie.
  • Fetch-Decode-Execute: 3-5 ciclu latență. Pipeline reduce throughput latență la 1 ciclu/instrucție.
  • Data hazard & branch misprediction: Pipeline stalls și flushes. Control processor performance.
  • PS5 3.5 GHz × 8 cores × 4+ IPC (instructions per cycle) = 100+ miliarde instrucții/sec potential.

Exercițiu Aplicat

🧠 Exercițiu: Performance Calculation cu Pipeline

Cerință: PS5 3.5 GHz. 4 instrucțiuni pipeline stages. 1% cache miss rate (latență 200 ciclu) și 5% branch misprediction rate (latență 10 ciclu).

  1. Calculează ciclu mediu per instrucție (CPI) fără hazard-uri
  2. Calculează impact cache misses pe CPI
  3. Calculează impact branch mispredictions pe CPI
  4. Total CPI cu hazard-uri
  5. IPC (instrucțiuni per ciclu) și throughput instrucțional
Arată rezolvarea

1. Base CPI (fără hazarduri): 1 ciclu per instrucție (pipeline perfect).

2. Cache miss impact: 1% instruc miss → 0.01 × (200 ciclu penalty per miss) = 2 ciclu penalty mediu.

3. Branch misprediction impact: 5% misprediction rate × 10 ciclu penalty = 0.5 ciclu penalty mediu.

4. Total CPI: 1 + 2 + 0.5 = 3.5 ciclu mediu per instrucție.

5. IPC & throughput: IPC = 1/CPI = 1/3.5 = 0.286 instruc/ciclu. Throughput: 3.5 GHz × 0.286 = 1 miliard instruc/sec (afront de 3.5 miliarde în perfect case!).

Impact performance: 3.5 CPI vs 1 CPI = 3.5× slowdown din ideal. Realist: 70% din teoric peak performance.
