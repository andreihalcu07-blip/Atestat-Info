Modul 4 — Arhitectură Hardware

Lecția 4.1 — CPU: De la Instrucțiune la Execuție

Nucleul calculelor. Cum ARM core PS5 preia o instrucție și o transformă în rezultat.

Începe Lecția

Introducere

CPU (Central Processing Unit): Execută instrucțiunile programului. Ce=? + Where=? + How=?. Citeaște cod, decodează, face operații, ține rezultate.

PS5: AMD Ryzen Zen 2 cu 8 ARM cores. Fiecare core = 3.5 GHz. Total 24 instrucții/ciclu (4 instrucții per core × 8 cores).

Instrucție exemplu: MOV, ADD, LOAD, STORE, JUMP, BRANCH. Toate reduse la microoperații în hardware.

Teorie Structurată

4.1 — Instrucție ARM64 Structure

Format 32-bit: Opcode (bits 31-25) | Operanzi (24-0). Exemplu: MOV x1, #0x1000.

Tipuri instrucții: Arithmetic (ADD, SUB), Logic (AND, OR), Memory (LOAD, STORE), Control flow (JUMP, BNE).

Encoding: Fiecare bit = info. Decoderul trebuie citi 32 biți și extrage: destinație registru, surse, opcode control.

4.2 — Execution Pipeline Detailat

Stage 1 (Fetch): PC + 4. Citire 4-byte instrucție din L1-I cache. Typical 3-cycle latență dacă cache hit.

Stage 2 (Decode): Parcurgere opcode. Selectare control semnale. Identificare registri sursa pe care ALU operează.

Stage 3 (Execute): ALU calculeaza. ADD, AND, OR, etc. Output latență 1-3 ciclu după input.

Stage 4 (Memory): Dacă LOAD/STORE: memorie I/O. Citire din cache L1-D. Altfel, bypass.

Stage 5 (Writeback): Rezultat → destinație registru via register file write port.

4.3 — Out-of-Order Execution

Concept: CPU nu execută exact ordinea instrucțiunilor. Planificator (scheduler) lansa instrucții când dependențe satisfăcute.

Exemplu: Instr1: LOAD x1, [x0] (latență 4 ciclu). Instr2: ADD x2, x2, x2 (independent). Scheduler face ADD pe ciclu 1 și Instr1 pe ciclu 2 (faster!).

Restriction: Nu poate schimba ordinea dacă data dependency. Rename register buffer ține track stări multiple per registru logic.

4.4 — Predicția Branch

Problem: BNE (branch-not-equal) instrucție. CPU nu stie rezultat CMP până pe ciclu 5. Ce fetch pe ciclu 2?

Predictors: History buffer ține track branch-uri trecute. Ghiceste direction. Daca greș, pipeline flush.

Tip modern: Correlated predictor (BTB = Branch Target Buffer). ~95% accuracy pe real code.

4.5 — Performance Metrics

IPC (Instructions Per Cycle): Câte instrucțiuni finalizare per ciclu clock. Ideal = 4-6 (PS5 Zen 2 spec), real = 2-3 (datele hazard-uri).

CPI (Cycle Per Instruction): Invers. CPI = 1/IPC. Daca IPC=2, CPI=0.5 ciclu/instr.

Cache misses, stalls, mispredictions: Reduc IPC. E job optimizatoare ca compilator să emita cod cu IPC mare.

Legătura Fizică – Informatică

Fizică: Tranzistori în ALU comutează la 3.5 GHz, realizând operații matematice în nanosecunde.

Informatică: Miliarde operații per secundă. Iluzia că programe curg "fluid".

Aplicare Directă

Exemplu: Fetch-Execute MOV x1, [x2+8]

Ciclu 1: Fetch MOV opcode din PC.

Ciclu 2: Decode. Scheduler vede LOAD dependency. Register x1 = rezultat. x2+8 = adresa calculata ALU.

Ciclu 3: ALU executa ADD x2+8. Output adresa memoria = 0x1008.

Ciclu 4: L1-D cache lookup adresa 0x1008. Cache hit. Data = 64-bit value.

Ciclu 5: Writeback. Data → x1.

Exemplu Real: Profiling Game Code PS5

Scenario: Game main loop. Citette liste de vertice (LOAD intens). Calculeaza positii. Rinde.

Issue: LOADs = cache misses din main memorie. Latență 200 ciclu. CPU stall. IPC cade la 0.5.

Optimizare: Prefetch înaintea loop. Load data în cache înainte de compute loop. IPC → 3-4. Performance +6× pe acel loop.

Probleme Frecvente

⚠️ De ce compiler optimizări importanțe?

Compilator ordoneaza instrucțiuni pentru a reduce hazard-uri și cache misses. Code layout afectează branch misprediction. Same algorithm = 2× speedup dacă scheduler optimizat.

⚠️ De ce out-of-order e greșitor pozibil?

Instruction reordering trebuie menține semantica program. Data race conditions dacă gresit. Multithread-ul trebuie atomic operații.

Recapitulare

  • Instrucție ARM64: 32-bit opcode + operanzi. CPU decoded → control semnale.
  • Pipeline: Fetch, Decode, Execute, Memory, Writeback. 5 stage latență.
  • Out-of-order: Scheduler lanseaza instrucțiuni independent paralel.
  • Branch prediction: Ghiceste direction. 95% accuracy, 10-cycle penalty misprediction.
  • IPC & CPI: Performance metric. IPC = instrucții/ciclu. Ideal 4-6, real 2-3 în practice.
  • Cache, prefetch, compiler optimizații controlează reală performance.

Exercițiu Aplicat

🧠 Exercițiu: Performance Estimate Loop

Cerință: 1000 iterații loop. Fiecare ster: 2 ADD, 1 LOAD (cache miss), 1 STORE. 3.5 GHz PS5.

  1. Ciclu per iterație (ideal)
  2. Ciclu per iterație cu cache miss latență 200
  3. Total timp macOS
  4. Optimizație: prefetch reduce miss rate 50%. Nou timp.
Arată rezolvarea

1. Ideal: 2 ADD + 1 LOAD (cache hit, ~3 ciclu) + 1 STORE (~1 ciclu) = 7 ciclu/iter.

2. Cu cache miss: 2 ADD (2 ciclu) + 1 LOAD miss (200 ciclu) + 1 STORE (1 ciclu) = 203 ciclu per iterație.

3. 1000 ste: 1000 × 203 = 203,000 ciclu. La 3.5 GHz: 203,000 / 3.5e9 = 58 microseconds.

4. Prefetch 50% hit rate: 2 ADD + 0.5× (200+3) + 0.5×3 + STORE = 2 + 101.5 + 1.5 + 1 = 106 cyclu per iter. 1000× = 106,000 ciclu = 30 microseconds. Speedup 2×!
